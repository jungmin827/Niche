## Review Priorities (in order)
1. Doc alignment — does this match `docs/` specs?
2. Architecture consistency — routers thin, services/repos separation, no logic leakage?
3. Maintainability — naming, structure, future readability
4. Naming clarity — explicit, precise, doc-aligned names
5. Unnecessary complexity — is there overengineering?
6. Product tone consistency — editorial, minimal, no flashy copy or UI

## Output Format
Separate your review into four sections:

**Good** — what is solid and should be kept
**Risky** — what could cause problems if left as-is
**Change now** — must fix before this is mergeable
**Can wait** — low-priority improvements for later

Be direct, specific, and technically grounded. Reference the relevant doc when flagging a violation.
