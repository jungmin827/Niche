# NichE Content Language v1

## 문서 목적
이 문서는 **NichE 앱 전반에서 사용하는 언어, 카피 톤, 마이크로카피, 명명 규칙**을 정의한다.
Codex, Cloud Code, 프론트엔드 에이전트, 기획자가 이 문서를 기준으로 UI 문구와 서비스 언어를 일관되게 구현할 수 있어야 한다.

이 문서는 다음을 다룬다.
- NichE가 어떤 말투로 사용자에게 말을 거는지
- 탭, 버튼, 상태, 빈 화면, 오류, 평가 코멘트의 기본 언어 규칙
- 어떤 표현을 피해야 하는지
- 영어 브랜드 단어와 한국어 UI 문장을 어떻게 섞을지

상위 기준 문서는 `niche_planning_v1.md`, `design_system.md`, `ai_quiz_generation.md`를 따른다.

---

## 1. 기본 언어 원칙

NichE의 언어는 친절해야 하지만 수다스럽지 않아야 한다.
사용자를 과하게 북돋우는 생산성 앱 말투도, 가벼운 SNS 말투도 피한다.
NichE의 언어는 **조용하고 절제된 에디토리얼 톤**이어야 한다.

핵심 원칙:
- 짧고 단정하게 쓴다.
- 감탄사를 남발하지 않는다.
- 이모지를 사용하지 않는다.
- 과한 동기부여 문구를 쓰지 않는다.
- 사용자를 판단하거나 압박하지 않는다.
- 설명보다 제안에 가깝게 쓴다.
- 결과를 과장하지 않는다.
- 사용자의 취향과 생각을 존중하는 언어를 쓴다.

한 줄 원칙:
> **Quiet, precise, editorial.**

## 2. 언어 전략

### 2.1 기본 언어
MVP의 기본 UI 언어는 **한국어**다.
대부분의 버튼, 안내 문구, 상태 문구, 오류 문구는 한국어로 제공한다.

### 2.2 영어 사용 규칙
영어는 브랜드 감도와 상징성을 위해 제한적으로 사용한다.
영어를 남발하지 않는다.

영어 사용 허용 범위:
- 랭크/칭호 이름 (`Surface`, `Focus`, `Signature` 등)
- 매우 짧은 보조 라벨
- 공유 템플릿에서의 타이포 요소
- 제품 브랜딩이 필요한 특정 헤더

영어를 기본 UI 문장 전체에 쓰지 않는다.
예:
- 탭명: `세션 / 피드 / 아카이브`
- 버튼: `시작하기`, `저장하기`, `다음으로`
- 보조 브랜딩: `Surface`, `Signature`

### 2.3 톤의 높낮이
NichE는 존댓말 기반으로 가되, 너무 상냥하거나 서비스업 말투처럼 쓰지 않는다.
기본 톤은 다음과 같다.
- 안내: 부드러운 존댓말
- 버튼: 짧은 명령형 또는 동사형 명사
- 상태: 서술형 단문
- 평가 코멘트: 단정한 피드백형 문장

예:
- 좋음: `기록을 남겨보세요.`
- 좋음: `세션이 완료되었어요.`
- 좋음: `조금 더 구체적으로 적어보면 좋아요.`
- 피해야 함: `와, 정말 대단해요!!!`
- 피해야 함: `당신의 엄청난 성장 여정을 응원합니다!`

## 3. 브랜드 무드 키워드

### 살아야 하는 인상
- 조용한 몰입
- 정제된 관찰
- 깊이 있는 취향
- 축적
- 기록
- 해석
- 아카이브

### 피해야 하는 인상
- 시험 앱
- 공부 인증 앱
- 생산성 코치 앱
- 과한 SNS 감탄 톤
- 챌린지/미션 앱 같은 문구

즉, NichE는 사용자를 채찍질하는 서비스가 아니라,
**취향을 기록하고 해석하게 만드는 조용한 프레임**이어야 한다.

## 4. 핵심 명명 규칙

## 4.1 탭 이름
고정:
- `세션`
- `피드`
- `아카이브`

변형하지 않는다.
예:
- `홈`으로 바꾸지 않음
- `커뮤니티`로 바꾸지 않음
- `마이`로 바꾸지 않음

## 4.2 아카이브 내부 언어
아카이브 안에는 두 축이 존재한다.

1. **블로그 글**
   - 라벨: `글`, `기록`, `포스트` 중 선택 가능하나 MVP 기본은 `글`
2. **세션 하이라이트**
   - 라벨: `하이라이트`

권장 구조:
- 섹션 제목: `글`
- 섹션 제목: `하이라이트`

## 4.3 세션 관련 용어
- Session bundle: 내부 구현 용어만 사용, 외부 UI에는 노출하지 않는다.
- 사용자 노출 용어는 `오늘의 세션`, `예정된 세션`, `진행한 세션`처럼 자연스럽게 쓴다.
- `번들`, `배치`, `워크플로우` 같은 시스템 용어는 숨긴다.

## 4.4 퀴즈 용어
내부 개발 문서에서는 `quiz`를 써도 되지만,
사용자에게는 `회고`, `질문`, `정리`, `돌아보기` 계열 표현을 우선 사용한다.

권장:
- `회고 질문`
- `오늘의 돌아보기`
- `세션을 마무리할 질문`

지양:
- `테스트 시작`
- `시험 보기`
- `채점 결과표`

---

## 5. 화면별 카피 원칙

## 5.1 세션 화면
세션은 앱에서 가장 핵심적인 화면이다.
말투는 집중을 방해하지 않아야 한다.

### 원칙
- 텍스트는 적게 쓴다.
- 사용자의 몰입을 깨는 감정 표현을 넣지 않는다.
- 시간, 주제, 기록 버튼만 또렷하게 둔다.

### 예시 카피
- `지금 보고 있는 것을 따라가 보세요.`
- `끝난 뒤 짧게 남기면 됩니다.`
- `15분 동안 한 가지에 머물러보세요.`

## 5.2 세션 완료 후 노트 입력
세션 직후는 사용자가 가장 솔직하게 적을 수 있는 순간이다.
질문은 넓게 열어두되 부담스럽지 않아야 한다.

권장 프롬프트:
- `무엇을 봤나요?`
- `어떤 점이 남았나요?`
- `짧게 정리해보세요.`
- `지금 떠오르는 생각을 적어보세요.`

지양:
- `학습 내용을 정확히 요약하세요.`
- `핵심 포인트를 빠짐없이 서술하세요.`

## 5.3 회고/퀴즈 화면
이 화면은 시험장이 아니라, 정리와 해석의 공간처럼 느껴져야 한다.

권장 헤더:
- `오늘의 회고`
- `세션을 마무리할 질문`
- `조금 더 깊게 적어보세요.`

문항 소개 문구 예시:
- `방금 지나온 시간을 자신의 말로 정리해보세요.`
- `기억에 남은 장면이나 생각을 적어보세요.`
- `당신만의 해석이 드러나도 괜찮아요.`

## 5.4 평가 결과 화면
점수는 보여주되 성적표처럼 느껴지면 안 된다.
피드백은 차갑지만 무심하지 않게 쓴다.

권장:
- `오늘의 점수`
- `이렇게 남았어요`
- `조금 더 선명하게 적을 수 있어요.`
- `관찰한 지점이 분명하게 드러납니다.`
- `생각의 방향이 잘 보입니다.`

지양:
- `정답률`
- `오답`
- `불합격`
- `재도전하세요`

## 5.5 피드
피드는 취향 교류의 공간이지만 과하게 떠들썩하면 안 된다.

권장 문구:
- `비슷한 결의 기록`
- `지금 올라온 글`
- `새로 저장된 하이라이트`

지양:
- `인기 급상승`
- `실시간 핫한 글`
- `지금 가장 뜨는 포스트`

## 5.6 아카이브
아카이브는 나의 세계가 축적되는 공간처럼 보여야 한다.

권장 문구:
- `쌓인 기록`
- `남겨둔 글`
- `저장한 하이라이트`
- `조용히 쌓여온 흔적`

---

## 6. 버튼 카피 규칙

버튼은 길지 않고 명확해야 한다.
버튼에서 브랜드 시를 쓰지 않는다.

### 기본 규칙
- 2~6자 중심
- 동사 우선
- 추상 표현보다 행동 표현 우선

### 권장 버튼 예시
- `시작하기`
- `기록하기`
- `저장하기`
- `다음으로`
- `완료하기`
- `공유하기`
- `나중에`
- `다시 보기`

### 상황별 권장
세션 시작:
- `세션 시작`

세션 노트 제출:
- `기록 저장`

회고 제출:
- `답변 제출`

하이라이트 저장:
- `하이라이트로 저장`

최종 이미지 저장:
- `이미지 저장`

피해야 하는 버튼 문구:
- `렛츠고`
- `도전하기`
- `성장 시작`
- `나를 업그레이드`

---

## 7. 빈 상태(Empty State) 원칙

NichE의 빈 상태 문구는 조용해야 한다.
억지로 행동을 유도하는 대신, 다음 행동을 조심스럽게 제안한다.

### 세션 빈 상태
- `아직 시작한 세션이 없어요.`
- `오늘의 첫 세션을 열어보세요.`

### 피드 빈 상태
- `아직 올라온 기록이 많지 않아요.`
- `조금 지나면 새로운 글과 하이라이트가 보일 거예요.`

### 아카이브 빈 상태
- `아직 남겨둔 기록이 없어요.`
- `세션을 마치고 글이나 하이라이트를 남겨보세요.`

### 블로그 빈 상태
- `아직 쓴 글이 없어요.`
- `오늘 스쳐간 생각부터 적어보세요.`

---

## 8. 오류/실패 문구 원칙

오류 문구는 기술적이어서는 안 되지만, 지나치게 감성적일 필요도 없다.
간단히 문제를 설명하고 다음 행동을 안내한다.

권장:
- `잠시 후 다시 시도해 주세요.`
- `저장 중 문제가 생겼어요.`
- `질문을 불러오지 못했어요.`
- `네트워크 상태를 확인해 주세요.`

지양:
- `알 수 없는 오류가 발생했습니다.`만 단독으로 노출
- `Oops!`
- `서버가 폭주했어요`

권장 형식:
- 문제 설명 한 줄
- 가능하면 행동 유도 한 줄

예:
- `질문을 준비하는 데 시간이 조금 더 필요해요.`
- `잠시 후 다시 확인해 주세요.`

---

## 9. 알림/토스트 문구

토스트는 매우 짧고 담백해야 한다.

예:
- `저장했어요.`
- `이미지를 저장했어요.`
- `하이라이트에 추가했어요.`
- `답변을 제출했어요.`
- `공유 링크를 복사했어요.`

과한 표현 금지:
- `완벽하게 저장되었어요!`
- `성공적으로 완료되었습니다!`

---

## 10. 랭크/칭호 언어 규칙

랭크 이름은 영어를 유지한다.
예:
- `Surface`
- `Trace`
- `Focus`
- `Depth`
- `Frame`
- `Tone`
- `Shape`
- `Archive`
- `Signature`
- `Canon`

하지만 설명 문구는 한국어로 쓴다.
예:
- `지금은 Focus에 머물러 있어요.`
- `기록이 쌓이며 다음 단계에 가까워졌어요.`

지양:
- `축하합니다! SSS급 취향 마스터 달성!`
- `당신은 진정한 고수입니다!`

권장:
- `지금의 결이 조금 더 선명해졌어요.`
- `쌓인 기록이 다음 단계로 이어지고 있어요.`

---

## 11. 공유 템플릿 카피 원칙

공유용 이미지는 브랜딩이 강해야 하지만, 문장은 더 절제되어야 한다.
텍스트가 많아지면 무드가 깨진다.

공유 카드에 적합한 요소:
- 세션 시간
- 짧은 제목
- 한 줄 문장
- 랭크/칭호
- 날짜

예:
- `45m of focus`
- `오늘의 기록`
- `남겨둔 장면`
- `Signature in progress`

MVP에서는 공유 이미지 내 텍스트를 짧게 유지한다.
2문장 이상 긴 문구는 지양한다.

---

## 12. 평가 코멘트 생성 규칙

AI 평가 코멘트는 제품 무드에 큰 영향을 준다.
다음 원칙을 따른다.

### 길이
- 질문별 코멘트: 1~2문장
- 전체 코멘트: 2~3문장

### 톤
- 건조하지만 무심하지 않게
- 지적보다 관찰에 가깝게
- 사용자의 표현을 존중하는 방향

### 좋은 예
- `무엇을 봤는지가 비교적 분명하게 드러납니다.`
- `개인적인 해석이 자연스럽게 이어집니다.`
- `조금 더 구체적인 장면이 들어가면 더 선명해질 수 있어요.`
- `느낀 점과 내용 요약이 잘 연결되어 있어요.`

### 피해야 하는 예
- `틀렸습니다.`
- `정답과 거리가 멉니다.`
- `더 열심히 복습해야 합니다.`
- `매우 우수합니다!!!`

## 13. Do / Don't

### Do
- 짧고 단정하게 쓴다.
- 과장 없이 제안한다.
- 사용자의 생각을 존중한다.
- 에디토리얼한 차분함을 유지한다.
- 영어는 상징적으로만 사용한다.

### Don't
- 과하게 응원하지 않는다.
- 시험/학습 앱 말투를 쓰지 않는다.
- SNS 감탄 톤을 쓰지 않는다.
- 이모지, 밈, 과장된 수사를 넣지 않는다.
- 기능 설명을 장황하게 쓰지 않는다.

---

## 14. 구현 메모

- 프론트엔드에서는 마이크로카피를 상수화해 관리한다.
- AI 생성 카피도 이 문서의 톤 규칙을 시스템 프롬프트에 반영한다.
- 버튼, 토스트, 빈 상태, 오류, 평가 코멘트는 가능하면 별도 리소스 파일로 분리한다.
- 브랜딩 영어 단어 목록은 별도 enum 또는 constants로 관리한다.

## 15. 최종 요약

NichE의 언어는 화려하지 않다.
