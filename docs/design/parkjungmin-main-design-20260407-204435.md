## Problem Statement

이미 깊은 관심사가 있는 사람들에게 그것을 기록하고, 깊이를 증명하고, 자연스럽게 공유할 수 있는 도구가 없다. 인스타그램은 표면적이고, 노션은 사적이며, Letterboxd는 영화 전용이다. 미누(微拿) 관심사를 위한 전용 기록-증명-공유 플랫폼이 존재하지 않는다.

## Demand Evidence

직접 수요 증거 없음 (아이디어 단계). 창업자 본인이 1차 사용자. 창업자가 직접 "기록하고 싶은데 제대로 된 도구가 없어서" 만들기 시작함.

시장 신호:
- Letterboxd MAU 2024년 상반기 55% 성장 — 틈새 소셜앱 수요 실재
- "지적 자본이 새로운 지위재" 트렌드 (2025년 분석) 확인
- 전용 "지식 과시/깊이 증명" 앱의 명확한 승자가 아직 없음

**다음 단계 (Assignment 참고):** 실제 수요 증거 수집 필요.

## Status Quo

타겟 사용자들이 현재 하는 것:
- Instagram, 블로그, Twitter에 단발적으로 공유
- 노션/옵시디언에 개인적으로 정리 (공유 안 됨)
- Letterboxd (영화), Goodreads (책) — 버티컬별로 분산

**문제:** 이 도구들 중 어느 것도 "얼마나 오래, 얼마나 깊이"를 시각적으로 증명하지 못함.

## Target User & Narrowest Wedge

**타겟:** 이미 깊은 관심사가 있지만 기록 도구가 마땅치 않은 사람.
하루 9시간 일하는 직장인, 퇴근 후 자신의 관심사를 기록하고 싶은데 인스타에 은근슬쩍 자랑하고 싶은 지적 허영심이 있는 사람.

**아직 아닌 것:** "관심사를 갖고 싶지만 시작을 못 하는 사람" (B타입) — 나중 확장.

**최소 웨지:** 관심사 1개 + 기록 몇 개 + 공유 카드 생성.
"나는 자연 와인을 4년째 파고 있다"를 인스타에 올릴 수 있는 카드를 만들어주는 것.

## Constraints

- Expo SDK 54 + React Native + TypeScript (현재 프론트엔드)
- FastAPI 백엔드 (현재 구조)
- Supabase Auth/Storage
- 혼자 개발 중
- 수익 없음, 사용자 없음

## Premises

1. 1차 타겟은 "이미 깊은 관심사가 있는 사람". B타입(관심사를 갖고 싶은 사람)은 나중.
2. 핵심 동기는 기록 + 정체성 표현 둘 다. 기록이 먼저, 공유는 자연스럽게.
3. 진짜 경쟁자는 노션, 개인 블로그, Letterboxd/Goodreads. 인스타그램 릴스가 아님.

**창업자가 Premise 1에 처음 이의를 제기했음:** "이미 관심사가 있는 사람"이라고 직접 정정함.
이는 타겟 선명도가 있다는 신호.

## Approaches Considered

### Approach A: 모든 것의 Letterboxd
Effort: S/M | Risk: Low
어떤 관심사든 로깅 + 공개 프로필. Letterboxd가 검증한 모델을 일반화.
단점: 각 버티컬이 이미 있음 (Goodreads, RateBeer 등). 차별화 어려움.

### Approach B: 깊이 증명 엔진 (Proof Engine) — 선택
Effort: M | Risk: Med
얼마나 오래, 얼마나 깊이 관심사에 몰두했는지를 시각적으로 증명하는 앱.
기록 + 타임라인 + 공유 카드. "은근슬쩍 인스타 자랑"을 직접 서비스.
**Letterboxd와의 차별점:** Letterboxd는 "무엇을 봤는가"만 기록한다. B는 "얼마나 오래, 얼마나 깊이"를 Depth 점수와 타임라인 바로 시각화한다. + 공유 카드 생성이 built-in.

### Approach C: 주간 회고 루프
Effort: M | Risk: Med/High
AI 기반 주간 회고 + 다이제스트. B타입도 포함하는 더 넓은 앱.
단점: AI 의존도 높음, 현재 백엔드와 맞지 않음, 콜드 스타트 어려움.

## Recommended Approach

**Approach B: 깊이 증명 엔진 (Proof Engine)**

핵심 루프:
1. 관심사 추가 (이름 + 시작 날짜)
2. 기록 추가 (텍스트 + 태그: 테이스팅 노트 / 독서 / 방문 / 관찰 등)
3. Depth 점수 자동 계산
4. 공유 카드 생성 (인스타 스토리 사이즈, 검정 배경)

**엔티티 설계 (MVP):**

```
Interest:
  id: uuid
  user_id: uuid
  name: string          -- 예: "자연 와인"
  started_at: date      -- 시작 날짜 (과거만 허용)
  is_public: bool       -- 기본 true (v1은 공개 전용)

Log (← 신규 테이블, BlogPost 구조 참고):
  id: uuid
  interest_id: uuid
  text: string (min 1자, max 2000자, 공백만 입력 불가)
  tag: enum(tasting_note | reading | visit | observation | other)
  logged_at: datetime
  is_public: bool       -- 기본 false (기록은 사적, 카드만 공개)
```

**Depth 점수 공식 (MVP):**
```
depth_score = log10(days_since_start + 1) × log10(record_count + 2)
```
- 예: 4년(1460일), 18개 기록 → log10(1461) × log10(20) ≈ 3.16 × 1.30 ≈ 4.1
- 소수점 1자리까지 표시
- v2에서 질적 평가(길이, 사진 첨부) 추가 가능 — MVP에서는 정량만

**공유 카드 스펙:**
- 크기: 540×960px (인스타 스토리 업로드 시 자동 업스케일) — react-native-view-shot으로 생성 (v1)
  - 375px 기준 View를 scale≈1.44로 캡처. 1080×1920 고해상도는 v2(서버사이드 렌더링)에서.
- 배경: #1a1a1a
- 포함 항목: 관심사명 / depth_score / 기간 / 기록 수 / 최근 기록 인용 / @username / "niche.app"

**백엔드 연결:**
- Interest → 신규 테이블 (현재 BlogPost와 별개)
- Log → 신규 테이블 (현재 BlogPost 구조 참고해 설계)
- Highlight → 공유 카드 생성 결과 저장에 재활용 가능
- "Session" 개념은 MVP에서 사용 안 함
- 공유 카드의 @username → Profile 테이블 (auth_user_id로 조회, 기존 구조 재활용)
- Depth 점수 계산: MVP는 API 요청 시 실시간 계산 (기록 수가 적으므로 캐싱 불필요)

**예외 처리:**
- 관심사 0개: 빈 화면 + "첫 관심사 추가하기" CTA
- 기록 0개: Depth 점수 표시 안 함, 타임라인 바 = 0%
- 시작 날짜 미래: 입력 불가 (오늘 이전만 허용)

**왜 이것인가:**
- "NichE"라는 이름 자체가 증명을 내포
- Letterboxd, 노션, 인스타 중 어느 것도 "얼마나 오래 파왔는지"를 보여주지 않음
- 공유 카드가 자연스러운 바이럴 루프를 만듦

**화면별 Depth 점수 표시:**
- Screen 1 (목록): 각 카드에 depth_score 표시 (예: "4.1 year depth")
- Screen 2 (상세): depth_score + days + record_count 수치 함께 표시
- 공유 카드: depth_score 포함

**Interest 수정/삭제 (MVP):**
- 수정: 이름, 시작 날짜 변경 가능 (Depth 점수 재계산)
- 삭제: MVP에서는 소프트 삭제. 연결된 Log는 cascade soft-delete. 복구 기능 없음.

**와이어프레임 (로컬 참고용):** `/tmp/niche-sketch.html`
- Screen 1: 관심사 목록 (카드 + 타임라인 바 + 최근 기록 미리보기)
- Screen 2: 관심사 상세 (Depth 지표 + 로그 + 공유 카드)

## Open Questions

1. **공개/비공개 — 결정됨:** Interest=public(기본 true), Log=private(기본 false), 공유 카드만 공개. MVP에서는 이 구조 고정.
2. **관심사 태그:** MVP는 자유형 문자열. v2에서 사전 정의 카테고리 추가 검토.
3. **소셜 레이어:** MVP는 단독 사용 (다른 유저 프로필 탐색 없음). v2 항목.
4. **Depth 점수 v2:** 기록 길이, 사진 첨부, 외부 링크 등 질적 평가 요소 추가 — MVP 이후.
5. **공유 카드 v2:** 서버사이드 렌더링 (더 일관된 렌더링) — react-native-view-shot 한계 해결 시점에 검토.

## Success Criteria

MVP 기준:
- 창업자 자신이 매일 사용하고 싶은 앱
- 관심사 추가 → 기록 → 공유 카드 생성이 3분 이내 가능
- 공유 카드를 실제로 인스타에 올리고 싶어지는 디자인

3개월 기준:
- 10명이 매주 기록하고 있음
- 그 중 3명 이상이 공유 카드를 인스타에 올림

## Distribution Plan

1단계: 창업자 본인 사용 (지금 당장)
2단계: 지인 10명 DM 초대 → 클로즈 베타 (Assignment 완료 후)
3단계: 공유 카드 자체가 배포 채널 — 인스타에 올리면 "niche.app" 워터마크 → 자연 유입

앱스토어 배포는 MVP 이후 별도 계획 수립.

## Dependencies

- 백엔드 스켈레톤 안정화 필요 (현재 진행 중 — CLAUDE.md 구현 우선순위 1)
- 프론트엔드 스켈레톤 안정화 (우선순위 2)
- Depth 점수 알고리즘 설계 (신규)
- 공유 카드 생성 기능 (신규 — MVP: react-native-view-shot 고정, v2: 서버사이드 렌더링 검토)

## The Assignment

이번 주 안에 할 일 하나: **실제 사람 5명에게 DM을 보내세요.**

"자연 와인 / 특정 음악 / 독립 영화 등" 당신이 아는 사람 중 깊은 관심사가 있는 사람에게:
"나 이런 앱 만드는데, 기록하고 공유하고 싶은 관심사 있어? 써볼 의향 있어?"

답이 "응" 이면 A타입 수요 증거가 생깁니다. "어 나는 그냥 인스타로도 충분한데" 이면 더 파야 합니다.

## What I noticed about how you think

- Q3에서 처음엔 일반적으로 답했다가, 추가 질문 후 "하루 9시간 일하는 직장인, 지적 허영심, 은근슬쩍 자랑" 이라는 구체적인 묘사를 꺼냈습니다. 첫 번째 답이 아니라 두 번째 답이 진짜였습니다. 이건 좋은 신호입니다 — 자신의 타겟 사용자를 실제로 알고 있다는 뜻.
- Premise 1을 즉각적으로 정정했습니다. "둘 다"라고 했다가 이내 "이미 관심사가 있는 사람"으로 좁혔어요. 창업자가 자기 제품의 경계를 알고 있다는 것입니다.
- NichE라는 이름 자체, "에디토리얼, 미니멀, 조용하고, 정확한" 이라는 톤 정의. 이건 취향입니다. 취향이 있는 사람이 만든 앱은 취향이 있는 사람들을 끌어당깁니다.
- 창업 동기가 "내가 직접 필요해서"입니다. 이게 맞습니다. 가장 좋은 제품은 창업자가 자신의 문제를 푸는 데서 시작합니다.
